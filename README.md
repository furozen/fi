# Финтех-платформа операционного мониторинга «Cabinet»

## 1. Цель проекта
Это специализированная платформа, предназначенная для управления данными.
Платформа проектируется как white-label решение и должна поддерживать несколько независимых партнерских брендов.

> ⚠️ **Критическое ограничение домена**: Платформа работает исключительно в режиме чтения данных мониторинга, разбора инцидентов и изменения конфигураций/статусов.

---

## 2. Формирование White Label Appearance (Highly Customized Concept)
Проект спроектирован как декларативный конструктор личных кабинетов. 
Кодовая база едина, но на этапе сборки (**Compile-time**) приложение меняет свое поведение, структуру и внешний вид для конкретного Партнера на основе файлов конфигурации:

* **Структурная кастомизация (Layout & Navigation)**: Лейауты (сейчас реализован `MinimalLayout`) изолированы в библиотеке `@cabinet/ui`. Они не содержат жестких ссылок на модули и строят интерфейс динамически. Неиспользуемые фичи полностью вырезаются из бандла на этапе сборки (**Tree Shaking**).
* **Локализация текстов (Languages & Overrides)**: Каждый проект имеет свой массив разрешенных языков. Система поддерживает глубокое слияние (Deep Merge) JSON-переводов: поверх базовых текстов ядра накатывается JSON-файл партнера, позволяя переопределять любые формулировки под бренд.
* **Визуальный стиль (Material 3 Design Tokens)**: Компоненты полностью лишены инлайнового хардкода цветов. Кастомизация внешнего вида (цвета, шрифты, скругления) реализуется через дизайн-токены **Angular Material 3** (системные CSS-переменные `--mat-sys-*`) и глобальные CSS-свойства (например, `--branding-border-radius`), подключаемые в конфигурации сборки конкретного клиента.
* **Сетевая стратегия (API Strategy)**: Точки входа работают через проксирующий `api-gateway` (задается в конфиге). Конкретные реализации подменяются через Angular DI на этапе сборки: ядро использует фасад `CabinetApiFacade` и токен `DEV_API_IMPLEMENTATION` (см. `libs/api/src/lib/cabinet-api.facade.ts`).


### White Label архитектура / брендинг


Примеры:

```text
WhiteStar
Acme
Default
```

#### Основные принципы

1. Бренд определяется на этапе сборки приложения.
2. Переключение бренда во время выполнения не поддерживается.
3. Каждый партнер имеет собственный набор конфигурации, провайдеров и визуального оформления.
4. Приложения (`operator`, `partner`) не зависят от конкретного бренда.

---

#### Структура каталогов

```text
brands/
├── index.ts               # Barrel: реэкспортирует типы и brandingConfig
├── brands.interface.ts    # Контракт BrandsConfigInterface
├── brand-config.ts        # Конфигурация бренда по умолчанию (brandingConfig)
├── theme.scss             # Тема по умолчанию
│
└── whitestar/             # Переопределение бренда WhiteStar
    ├── brand-config.ts    # Конфигурация WhiteStar (fileReplacements → brand-config.ts)
    └── theme.scss         # Тема WhiteStar (подключается через массив styles)
```

> Файлы `providers.ts` и директория `default/`, упоминавшиеся в ранних версиях документа, в текущей структуре отсутствуют — переопределение по бренду реализовано через `brand-config.ts` и `theme.scss`.

---

#### Назначение файлов

##### brand-config.ts

Содержит данные бренда (`brandingConfig: BrandsConfigInterface`):

* название компании (`name`);
* логотип (`logo`);
* тип лейаута (`layoutType`: `'minimal' | 'sidebar'`);
* список поддерживаемых языков (`supportedLanguages`);
* флаги доступности фич (`enabledFeatures`).

##### theme.scss

Содержит визуальное оформление бренда:

* Material 3 theme;
* CSS custom properties;
* design tokens;
* фирменные стили.

#### Подключение бренда

Подключение бренда выполняется через конфигурацию сборки Nx (см. `apps/operator/project.json`, секция `configurations.whitestar`).

Для бренда WhiteStar:

* файл `brands/brand-config.ts` подменяется на `brands/whitestar/brand-config.ts` (через `fileReplacements`);
* тема `brands/whitestar/theme.scss` подключается через массив `styles` вместо `brands/theme.scss`.

Таким образом приложение всегда работает с едиными точками входа (`@cabinet/brands`) и не содержит прямых ссылок на конкретного партнера.

### White Label архитектура / Backend(API) реализации

API-реализация рассматривается как отдельная ось конфигурации и не является частью бренда.

## 3. Технологический стек
* **Фреймворк:** Angular 22 (Стабильный релиз).
* **Рантайм-стандарты:** Полноценный **Zoneless** (работа без `zone.js`), **Angular Signals**, **Signal Forms** (`@angular/forms`), новый управляющий синтаксис (`@if`, `@for`).
* **Монорепозиторий:** Nx (Project Crystal / Inferred Targets) с единым корневым `node_modules`.
* **Сборщик:** Application Builder на базе **Esbuild** / Vite.
* **UI-фреймворк:** Angular Material 3 + SCSS.
* **Тестирование:** **Vitest** (через `@analogjs/vite-plugin-angular` для сверхбыстрого тестирования библиотек-компонентов на лету без предварительной сборки).

## 4. Структура проекта 

```text
├── apps/                          # ТОЧКИ СБОРКИ (Приложения)
│   ├── operator/                  # Основной кастомизируемый ЛК для партнеров и операторов
│   └── developer/                 # Технический ЛК для разработчиков (детальный аудит и контроль)
│
├── libs/                          # ИЗОЛИРОВАННЫЕ БИБЛИОТЕКИ
│   ├── api/                       # Ядро сетевого взаимодействия (Фасады, контракты, моки)
│   │   ├── src/lib/concrete/      # Реализации (BasePartnerApiService, DevApiService, MockApiService)
│   │   ├── src/lib/interfaces/    # Контракты (ICabinetApi, IDevCabinetApi, IApiLogs)
│   │   └── src/lib/mocks/         # Изолированные типизированные моковые данные (partners.mock.ts, payments.mock.ts)
│   ├── shared/                    # Чистые утилиты, хелперы, типы конфигураций (без циклических связей)
│   ├── ui/                        # Дизайн-система (Шаблон MinimalLayout на токенах M3)
│   └── features/                  # Функциональные модули бизнес-логики (Изолированные либы)
│       ├── auth/                  # Модуль авторизации на Signal Forms (Компонент Login)
│       ├── payments/              # Интерфейсы отслеживания затыков в платежах
│       ├── partners/              # Управление лимитами и комиссиями партнеров
│       ├── operation-audit/       # Детальный аудит операций (для ЛК Developer)
│       └── rmq-monitor/           # Мониторинг RabbitMQ
```
