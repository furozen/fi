## Architecture Decision Record (ADR)## ADR-0003: Внедрение событийно-ориентированной шины данных для децентрализации модулей## Статус
Принято (Accepted) — 2026-06-23
## Контекст
Платформа Cabinet строится по принципу Highly Customized White Label монорепозитория, где бизнес-фичи (payments, partners, stats) оформлены в виде изолированных, независимых библиотек (libs/features/*).
При классическом подходе к проектированию Angular-приложений модули вынуждены импортировать сервисы друг друга или завязываться на жесткую иерархию родительских и дочерних компонентов (@Input / @Output).
Это приводит к следующим критическим проблемам:

1. Жесткая связанность (Tight Coupling): Компонент мониторинга платежей не может быть скомпилирован без модуля управления партнерами, если ему нужны данные о лимитах этого партнера.
2. Нарушение Tree Shaking: Сборщик Esbuild не может исключить неиспользуемую фичу из White Label бандла конкретного клиента, так как на неё есть неявные перекрестные ссылки из других слоев кодовой базы.
3. Сложность интеграции с асинхронными источниками: В будущем платформе потребуется поддержка WebSockets и Server-Sent Events (SSE) для вывода затыков в транзакциях реального времени. Привязка к классическим HTTP-сервисам усложнит этот переход.

## Решение
Внедрить в ядро системы централизованный слабосвязанный брокер событий — DataBusService на базе RxJS потоков (Subject), разместив его в слое libs/shared в качестве глобального синглтона.
Вся коммуникация между изолированными фичами, а также между инфраструктурными слоями (сеть/сокеты) и UI-компонентами переводится на декларативный обмен сообщениями строго по контракту IDataBusMessage.
## Ключевые правила маршрутизации (Route):

* Route.Inner: Внутренние события фронтенда. Используются для мгновенной синхронизации стейта между независимыми UI-виджетами на экране (например, изменение настроек фильтра в одном компоненте должно обновить таблицу в другом).
* Route.Inward: Входящий поток данных. Сюда сокет-интерцепторы или API-сервисы выбрасывают сообщения о событиях внешнего мира (обнаружен затык операции на шлюзе, упал баланс партнера).
* Route.Outward: Исходящий поток. Команды интерфейса, адресованные внешним системам или бэкенд-службам.

## Последствия## Положительные (Pros):

1. Тотальная децентрализация: Фичи payments и partners теперь знают только о контракте шины IDataBusMessage. Они полностью изолированы друг от друга и могут разрабатываться параллельными командами OpenCode автономно.
2. Чистый Tree Shaking: Если конкретному White Label партнеру отключена фича stats (аналитика), компилятор Esbuild полностью вырежет её бандл из сборки. Отсутствие прямых связей между компонентами гарантирует бесконфликтное удаление мертвого кода.
3. Бесшовная Zoneless-совместимость (Angular 22): Потоки шины данных легко конвертируются на уровне компонентов в нативные Сигналы через toSignal(dataBus.inner$), исключая необходимость в zone.js и ручных вызовах ChangeDetectorRef.
4. Готовность к масштабированию (SSE/WebSockets): Подключение реального бэкенд-стриминга логов сведется к написанию одного сервиса, который будет слушать сокет и генерировать сообщения в шину с типом Route.Inward.

## Отрицательные (Cons):

1. Сложность трассировки: Потоки событий сложнее отлаживать в консоли браузера, чем прямые вызовы методов. Для компенсации этого в конструкторе DataBusService для дев-окружения (development) должен быть предусмотрен сквозной логгер, выводящий тип и тело каждого сообщения.
2. Раздувание перечислений: Перечисление DataBusMessageType станет единой точкой сборки всех типов событий системы и будет расти по мере добавления бизнес-требований.
