## Architecture Decision Record (ADR)## ADR-0004: Внедрение подсистемы контекстного цветного логирования с шиной агрегации## Статус
Принято (Accepted) — 2026-06-23
## Контекст
В крупном монорепозитории Cabinet, состоящем из множества изолированных библиотек-фич (libs/features/*) и независимых точек сборки (apps/*), стандартный отладочный вывод через хаотичные console.log приводит к следующим проблемам:

1. Потеря контекста (Context Pollution): В общей консоли браузера невозможно быстро определить, какой именно слой (ядро API, общая шина данных или конкретный UI-компонент) сгенерировал строку или выбросил предупреждение.
2. Сложность разбора инцидентов (Trace Hardness): Потоковые события (на базе DataBusService) сливаются в нечитаемый текстовый массив, что увеличивает время локализации «затыков» в транзакциях.
3. Отсутствие рантайм-агрегации: Инженерный ЛК Разработчика (apps/developer) и ЛК Поддержки (apps/partner) требуют вывода системных логов прямо на UI-экран админ-панели в реальном времени. Стандартный console.log перехватить на уровне Angular-компонентов без костылей невозможно.
4. Утечки памяти в Zoneless: Неконтролируемое логирование тяжелых объектов в долгоживущих RxJS-потоках Angular 22 (без zone.js) может приводить к удержанию ссылок сборщиком мусора (Garbage Collector).

## Решение
Внедрить кастомную подсистему изолированного контекстного логирования на базе фабрики createLogger(context), разместив её в слое libs/shared/src/lib/utils/.
## Ключевые архитектурные правила подсистемы:

1. Цветовое кодирование по Хэшу (Color by Context): Функция createLogger автоматически рассчитывает пастельный цвет фона для тега контекста на основе алгоритма getColorByWordPRand(context). Каждая библиотека получает свой уникальный, персистентный визуальный маркер в консоли (например, [DataBusService] всегда бирюзовый, [api] — синий).
2. Compile-time / Runtime фильтрация уровней логов: Подсистема поддерживает 6 уровней логирования (verbose, log, info, debug, warn, error). Допустимый уровень вывода управляется декларативным флагом из конфигурации текущего скомпилированного White Label бренда (brandingConfig).
3. Глобальный стрим агрегации (logging$): Любой вызов метода логгера форматирует сообщение (строка + сериализованные объекты через JSON.stringify) и отправляет его в глобальный горячий поток Subject.
4. Управление ссылками (GC Safety): Внутри оператора вывода все аргументы принудительно очищаются (args = null), гарантируя своевременное освобождение памяти в Zoneless-окружении.

## Последствия## Положительные (Pros):

1. Сквозная визуальная трассировка инцидентов: Консоль становится структурированной. Разработчик OpenCode по цветам тегов может за секунду восстановить цепочку: Событие в UI ──► Трансляция в шину DataBus ──► Сетевой запрос в API.
2. Готовность к экрану «Live Logs»: Наличие потока logging$ позволяет одной строчкой кода вывести консоль внутрь Angular-компонентов в ЛК Разработчика/Саппорта (через toSignal(logging$)), реализуя полноценный интерфейс мониторинга «затыков».
3. Изоляция сред выполнения: На продакшене White Label бандла (production) уровень логов автоматически выставляется в error или warn, полностью очищая консоль конечного клиента от отладочной информации и защищая ПДн.

## Отрицательные (Cons):

1. Нагрузка на сериализацию: Вызовы логирования с тяжелыми циклическими объектами (Circular Structures) внутри tail-аргументов будут обрезаться. Разработчики обязаны передавать в логи только атомарные данные или использовать безопасный хелпер serializeObj.
2. Запрет на прямой console.log: Отныне использование прямого метода console.log() в кодовой базе монорепозитория Cabinet считается архитектурным нарушением и должно блокироваться на уровне правил ESLint.
